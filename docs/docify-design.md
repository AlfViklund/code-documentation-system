# docify — проектный документ

> Живая документация фич проекта, привязанная к коду, для людей и кодинг-агентов.
> Аналог graphify/codegraph, но вместо графа кода — **граф фич и документации**, с автоматической подсветкой устаревших доков при изменении кода.

- **Статус документа:** design doc v1
- **Форма продукта:** Python-пакет (`pip install docify`) — CLI + MCP-сервер + локальный веб-дашборд
- **Хранение данных:** в репозитории пользователя (`docs/features/*.md` + производный индекс `.docify/`)

---

## 1. Обзор и мотивация

### 1.1 Проблема

Современная разработка всё чаще ведётся через кодинг-агентов (Claude Code, Cursor, v0 и т.п.). При этом:

1. **Агенты «забывают» фичи.** У агента нет долговременной памяти о проекте: какие фичи существуют, зачем они сделаны, какие решения были приняты. Каждая сессия начинается «с нуля» или с дорогого повторного исследования кодовой базы.
2. **Документация гниёт.** Доки пишутся один раз и не обновляются. Никто не знает, каким докам можно верить. Агент, прочитавший устаревшую доку, делает неверные изменения.
3. **ТЗ теряются.** Человек кидает агенту ТЗ в чат — агент реализует часть, остальное растворяется в истории переписки. Через месяц никто не помнит, что было запланировано, а что реализовано.
4. **Техдолг и планы не структурированы.** «Надо будет переделать» живёт в головах, TODO-комментариях и закрытых чатах.

### 1.2 Решение

**docify** — инструмент, который делает документацию фич первоклассной сущностью репозитория:

- каждая **фича** описана в отдельном markdown-файле и **привязана к конкретному коду** (файлам и символам);
- при изменении привязанного кода фича автоматически помечается как **требующая обновления документации** (подсветка stale);
- агенты работают с фичами через **MCP-сервер**: узнают, какие фичи затрагивает изменяемый код, обновляют доки, внедряют ТЗ;
- человек видит состояние проекта в **веб-дашборде**: карта фич, красные устаревшие доки, бэклог техдолга и развития.

### 1.3 Позиционирование относительно graphify / codegraph

| | graphify / codegraph | docify |
|---|---|---|
| Что индексирует | Структуру кода (символы, вызовы, зависимости) | **Фичи, документацию, ТЗ, бэклог** |
| Отвечает на вопрос | «Как устроен код? Что от чего зависит?» | «**Зачем** этот код? Какие фичи он реализует? Актуальны ли доки?» |
| Источник истины | Сам код | Документация, привязанная к коду |
| Отслеживание изменений | Инкрементальная переиндексация кода | Инкрементальная проверка **устаревания доков** относительно кода |

docify **дополняет** graphify/codegraph, а не заменяет: graphify даёт агенту карту «как», docify — карту «что и зачем». Модель установки и UX сознательно повторяют graphify: `pip install`, команда `install --project` для подключения MCP к агентам, данные живут в репо.

## 2. Ключевые концепции и модель данных

### 2.1 Feature (фича)

Единица документации. Хранится как `docs/features/<id>.md`.

| Поле | Тип | Описание |
|---|---|---|
| `id` | slug | Уникальный идентификатор, совпадает с именем файла (`auth-login`) |
| `title` | string | Человекочитаемое название |
| `status` | enum | `planned` \| `in-progress` \| `implemented` \| `needs-update` \| `deprecated` |
| `priority` | enum | `p0` \| `p1` \| `p2` \| `p3` |
| `tags` | string[] | Произвольные теги (`auth`, `api`, `ui`) |
| `anchors` | Anchor[] | Привязки к коду (см. 2.2) |
| `relations` | Relation[] | Связи с другими фичами: `depends-on`, `part-of`, `supersedes` |
| `specs` | SpecRef[] | Ссылки на внедрённые ТЗ (см. 2.3) |
| `updated_at` | ISO date | Когда доки последний раз подтверждались актуальными |
| `verified_commit` | sha | Коммит, на котором доки были зафиксированы как актуальные |
| body | Markdown | Свободный текст: описание, поведение, edge cases, принятые решения |

Жизненный цикл статусов:

```
planned ──► in-progress ──► implemented ◄──┐
                                │          │
                                ▼          │ (агент/человек обновил доки:
                          needs-update ────┘  mark_updated)
                                │
                                ▼
                           deprecated
```

- `needs-update` выставляется **автоматически** чекером (код якоря изменился) либо **вручную/агентом** (ТЗ меняет фичу).
- `planned` — фича описана, но кода нет (нет якорей или якоря помечены как ожидаемые).

### 2.2 Anchor (привязка к коду)

Одна фича → N якорей. Два типа:

**file-anchor** — привязка к файлу целиком:

```yaml
- type: file
  path: src/auth/login.ts
  blob: 8f3a91c...        # git blob hash файла на момент verified_commit
```

**symbol-anchor** — привязка к символу (функция/класс/метод) через tree-sitter:

```yaml
- type: symbol
  path: src/auth/login.ts
  symbol: LoginService.authenticate   # qualified name
  kind: method
  body_hash: 4be2d07...   # sha256 нормализованного тела символа (без пробелов/комментариев)
```

Symbol-anchor устойчивее к шуму: изменение соседнего кода в том же файле не помечает фичу как stale — только изменение самого символа.

### 2.3 Spec (ТЗ)

Входящее требование, внедрённое в документацию агентом. Хранится как `docs/specs/<id>.md` с frontmatter:

```yaml
id: spec-2026-07-oauth
title: Добавить OAuth-провайдеры
received_at: 2026-07-18
source: "ТЗ от продакта, чат #product"
features:            # раскладка ТЗ по фичам
  - id: auth-login
    action: update   # фича существует, требует изменения → needs-update
  - id: auth-oauth
    action: create   # новая фича → planned
status: open         # open | in-progress | done
```

### 2.4 Backlog item

Элемент техдолга или плана развития. Хранится в `docs/backlog/<id>.md`:

```yaml
id: bl-042
title: Вынести rate-limiting в middleware
type: debt           # debt | growth
priority: p2
features: [api-rate-limit]
status: open         # open | in-progress | done | wontfix
```

### 2.5 Хранение

```
repo/
├── docs/
│   ├── features/          # источник истины — человекочитаемые .md, дифаются в PR
│   │   ├── auth-login.md
│   │   └── ...
│   ├── specs/             # внедрённые ТЗ
│   └── backlog/           # техдолг и развитие
└── .docify/
    ├── config.yaml        # настройки: языки, игнорируемые пути, режим строгости
    └── index.json         # производный кэш: хэши, якоря, last_indexed_commit
                           # (в .gitignore ИЛИ коммитится — настраивается; пересобирается всегда)
```

Принципы: **всё в git** (переживает ветки, ревьюится в PR), markdown + YAML frontmatter (читается человеком и парсится машиной), индекс — только кэш, его потеря не теряет данных.

## 3. Механизм отслеживания устаревания (ядро)

### 3.1 Алгоритм `docify check`

1. Определить изменённые файлы с момента `last_indexed_commit`:
   `git diff --name-only <last_indexed_commit>..HEAD` + незакоммиченные изменения (`git status`).
2. Для каждой фичи, чьи якоря пересекаются с изменёнными файлами:
   - **file-anchor:** сравнить текущий git blob hash с сохранённым.
   - **symbol-anchor:** распарсить файл tree-sitter'ом, найти символ по qualified name, вычислить хэш нормализованного тела, сравнить с сохранённым.
3. Присвоить результат по каждому якорю и агрегировать в статус фичи.

### 3.2 Статусы результата проверки

| Статус | Условие | Действие |
|---|---|---|
| `fresh` | Все якоря совпадают | — |
| `stale` | Хотя бы один якорь: код изменился | Фича → `needs-update`, подсветка в дашборде/CLI/MCP |
| `broken` | Якорь исчез: файл удалён или символ не найден | Подсветка красным, требуется ре-привязка |
| `unimplemented` | Фича `planned`, якорей нет | Подсветка в бэклоге «не реализовано» |

### 3.3 Устойчивость к переименованиям

- **Файлы:** `git diff -M` (rename detection) — если файл переименован без изменений, якорь ре-маппится автоматически, фича остаётся `fresh`.
- **Символы:** если символ не найден по старому имени, но в файле есть символ с идентичным `body_hash` — предлагается авто-ремап (`docify check --fix-anchors` применяет).
- **Нормализация тела символа** (удаление пробельного шума и комментариев перед хэшированием) исключает ложные stale от форматтеров.

### 3.4 Инкрементальность

Пересчёт только для файлов, изменённых с `last_indexed_commit`. Полный переиндекс (`docify index --full`) нужен только при первом запуске или смене конфига. На репозитории в 100k+ файлов проверка типичного диффа — доли секунды.

### 3.5 Интеграции

- **`docify check --ci`** — exit code ≠ 0, если PR меняет код фичи без обновления её доков; markdown-отчёт для комментария в PR.
- **git hooks:** `pre-commit` (предупреждение), `post-merge` (переиндексация).
- Опционально: GitHub Action, публикующий отчёт «затронутые фичи» в PR.

## 4. CLI

```
docify init                          # создать docs/features, .docify/config.yaml, проиндексировать
docify index [--full]                # (пере)построить индекс якорей
docify check [--ci] [--fix-anchors]  # проверить устаревание, отчёт
docify feature add <id> [--title --status --tags]
docify feature list [--status stale|planned|...] [--tag <tag>]
docify feature show <id>             # доки + состояние якорей + diff изменений
docify feature update <id> --status implemented   # и др. поля
docify link <feature-id> <path>[::<symbol>]       # добавить якорь
docify mark-updated <feature-id>     # зафиксировать доки как актуальные (обновить хэши)
docify spec ingest <file|- >         # заготовка: создать spec, разложить по фичам (интерактивно/агентом)
docify backlog [add|list|done] ...
docify serve [--port 4321]           # веб-дашборд
docify mcp                           # запустить MCP-сервер (stdio)
docify install --project             # прописать MCP в конфиги агентов (Claude Code, Cursor, ...)
```

Вывод `check` в терминале — таблица с цветовой индикацией: зелёный `fresh`, жёлтый `stale`, красный `broken`, серый `planned`.

## 5. MCP-сервер — главная ценность для агентов

### 5.1 Инструменты (tools)

**Чтение:**

| Tool | Назначение |
|---|---|
| `list_features(status?, tag?, query?)` | Обзор всех фич — агент «вспоминает» проект в начале сессии |
| `get_feature(id)` | Полные доки + якоря + связанные спеки и бэклог + состояние актуальности |
| `find_features_for_code(path, symbol?)` | **Обратный поиск:** какие фичи затрагивает файл/символ, который агент собирается менять |
| `check_staleness(scope?)` | Что требует обновления доков (весь проект или подмножество путей) |
| `get_backlog(type?: debt\|growth, status?)` | План доработок — агент видит техдолг и roadmap |
| `get_spec(id)` / `list_specs(status?)` | Входящие ТЗ и их статус внедрения |

**Запись:**

| Tool | Назначение |
|---|---|
| `upsert_feature(id, fields, body?)` | Создать/обновить фичу |
| `link_code(feature_id, anchors[])` | Привязать код к фиче |
| `mark_updated(feature_id)` | «Доки актуальны»: пересчитать и сохранить хэши якорей, снять `needs-update` |
| `ingest_spec(text, source?)` | Внедрить ТЗ: создать spec-файл, разложить по фичам (новые → `planned`, изменяемые → `needs-update` с описанием требуемых изменений) |
| `upsert_backlog_item(...)` | Управление бэклогом |

### 5.2 Ресурсы и prompts

- **Resource `docify://overview`** — компактная сводка проекта: список фич со статусами (дёшево по токенам, для начала сессии).
- **Prompt «workflow»** — встроенная инструкция агенту:

> 1. Перед изменением кода вызови `find_features_for_code` для затрагиваемых файлов.
> 2. Прочитай доки затронутых фич через `get_feature` — там контекст и принятые решения.
> 3. После изменения кода обнови тело доков и вызови `mark_updated`.
> 4. Если реализуешь ТЗ — сначала `ingest_spec`, затем работай по разложенным фичам.
> 5. Обнаружил техдолг — зафиксируй через `upsert_backlog_item`, не теряй.

`docify install --project` дополнительно добавляет эту инструкцию в `CLAUDE.md` / rules-файлы агентов, чтобы workflow соблюдался без напоминаний.

## 6. Рабочие сценарии

### 6.1 Агент меняет код

1. Агент получает задачу «поменять логику логина».
2. `find_features_for_code("src/auth/login.ts")` → фича `auth-login` (+ связанная `auth-session`).
3. `get_feature("auth-login")` → агент читает поведение, edge cases, прошлые решения — **не изобретает заново и не ломает неочевидное**.
4. Агент меняет код, правит тело доки фичи, вызывает `mark_updated("auth-login")`.
5. Доки остаются зелёными; CI-проверка проходит.

### 6.2 Человек кидает ТЗ

1. Человек: «Вот ТЗ на OAuth» (текст/файл).
2. Агент: `ingest_spec(text)` → создан `docs/specs/spec-2026-07-oauth.md`; фича `auth-oauth` создана со статусом `planned`; фича `auth-login` помечена `needs-update` с пояснением, что именно меняется.
3. Нереализованное **подсвечено** в дашборде и в `list_features(status="planned")` — ТЗ не потеряется, даже если реализация отложена.
4. Когда агент реализует — статусы переводятся в `implemented`, spec закрывается.

### 6.3 Разработчик открывает дашборд

`docify serve` → карта фич: зелёные актуальные, жёлтые stale, красные broken, серые planned. Клик по фиче → доки + diff «что изменилось в коде с последней фиксации». Вкладка бэклога → техдолг и планы развития.

### 6.4 CI ловит расхождение

PR меняет `src/auth/login.ts`, но доки `auth-login` не обновлены → `docify check --ci` падает, в PR появляется отчёт: «Изменение затрагивает фичи: auth-login (stale). Обновите документацию или выполните mark-updated».

## 7. Веб-дашборд (`docify serve`)

**Для людей, read-mostly** (правки — через CLI/MCP/файлы, чтобы источником истины оставался git).

Экраны:

1. **Карта фич** — сетка/список карточек с цветовой индикацией статуса, фильтры по статусу/тегам/приоритету, поиск. Сводка сверху: «42 фичи: 35 fresh, 4 stale, 1 broken, 2 planned».
2. **Страница фичи** — отрендеренные доки; список якорей с состоянием; для stale-якорей — diff кода «зафиксированное ↔ текущее»; история изменений (из git log по файлу фичи); связанные спеки и бэклог.
3. **Бэклог** — две колонки-доски: техдолг / развитие, со статусами и связями с фичами.
4. **Входящие ТЗ** — список спеков, прогресс внедрения по фичам.

Технически: FastAPI + uvicorn внутри Python-пакета, отдаёт собранную статическую SPA (React + Vite, собирается при релизе пакета); API читает `docs/` и `.docify/index.json`; websocket для live-обновления при изменении файлов.

## 8. Формат файла фичи — спецификация

`docs/features/auth-login.md`:

```markdown
---
id: auth-login
title: Вход по email и паролю
status: implemented
priority: p0
tags: [auth, api]
updated_at: 2026-07-10
verified_commit: a1b2c3d
anchors:
  - type: symbol
    path: src/auth/login.ts
    symbol: LoginService.authenticate
    kind: method
    body_hash: 4be2d07f...
  - type: symbol
    path: src/auth/session.ts
    symbol: createSession
    kind: function
    body_hash: 91cc03aa...
  - type: file
    path: src/app/api/auth/login/route.ts
    blob: 8f3a91c2...
relations:
  - type: depends-on
    id: auth-session
specs:
  - spec-2026-07-oauth
---

## Что делает

Аутентификация пользователя по email + паролю. Возвращает httpOnly session cookie.

## Поведение

- Пароли: bcrypt, cost 12.
- 5 неудачных попыток → блокировка на 15 минут (см. `api-rate-limit`).
- Сессия живёт 30 дней, продлевается при активности.

## Edge cases

- Email нормализуется в lowercase до поиска.
- При смене пароля инвалидируются все сессии, кроме текущей.

## Принятые решения

- 2026-05: отказались от JWT в пользу серверных сессий — нужен мгновенный logout.
```

Frontmatter валидируется схемой (pydantic); `docify check` ругается на невалидные файлы.

## 9. Архитектура пакета

```
docify/
├── core/          # модели (pydantic), чтение/запись feature/spec/backlog файлов, index.json
├── anchors/       # git-операции (blob hash, diff, rename detection) + tree-sitter парсинг символов
├── checker/       # алгоритм проверки устаревания, агрегация статусов, ремап якорей
├── cli/           # typer-приложение
├── mcp/           # MCP-сервер (официальный python mcp SDK, stdio transport)
└── web/           # FastAPI-приложение + собранная SPA-статика
```

- **Языки на старте:** Python, TypeScript/JavaScript (+TSX/JSX), Go. Расширение — подключением tree-sitter грамматики + маппер «узел → qualified symbol name» (~150 строк на язык).
- **Зависимости:** `gitpython`/subprocess-git, `tree_sitter` + language packs, `typer`, `pydantic`, `mcp`, `fastapi`, `uvicorn`.
- Файлы вне поддерживаемых языков привязываются file-anchor'ами — деградация мягкая.

## 10. Roadmap docify

| Версия | Содержание |
|---|---|
| **MVP (v0.1)** | core + формат файлов + CLI: `init`, `index`, `check`, `feature`, `link`, `mark-updated` |

| **v0.4** | LLM-оценка значимости изменений (опциональный шаг: «изменение косметическое → не помечать stale»); авто-черновики обновлений доков по diff |
| **Идеи** | Интеграция с graphify-графом (авто-предложение якорей по графу зависимостей); GitHub App; командный режим с общим сервером; экспорт в Docusaurus/Mintlify |
